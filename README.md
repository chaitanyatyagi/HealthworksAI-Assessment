## Overview

This project processes insurance **Evidence of Coverage (EOC)** PDF documents by extracting structured data from Chapter 4 (typically covering benefits, copays, and coverage details) using an LLM-powered pipeline. The extracted data is stored as a CSV and made queryable through a conversational chatbot interface. Users can also upload their own CSV files and query those directly.

## Modules

### `app.py` — Main Application

The entry point; builds and orchestrates the entire Streamlit UI.

**Sections:**

**1. Pipeline Section**
- Scans the `./EOC` folder for PDF files.
- On "Run Pipeline" click, calls `run_batch_pipeline()`, which:
  - Spawns a background timer thread (using `threading` + Streamlit's `add_script_run_ctx`) to display live elapsed time.
  - Iterates over each PDF, calling `process_single_file()`.
  - Shows per-file status badges (pending / running / done) in real time.
  - Concatenates all resulting DataFrames and saves to `output/output.csv`.
- Displays post-run metrics: files processed, total rows, column count, and elapsed time.
- A "Delete Output" button clears the saved CSV and resets session state.

**2. Data Context Section**
- Lets users select which data source the chatbot should query:
  - **Pipeline output** — the CSV generated by the batch pipeline.
  - **csv upload** — a CSV uploaded by the user via the file uploader widget.
  - **None** — disables the chat input entirely.
- Renders a live preview of the selected dataset using `st.dataframe`.
- Offers a download button for the pipeline output CSV.

**3. Assistant (Chatbot) Section**
- Renders a scrollable chat history with distinct user/bot bubble styles.
- Uses `st.chat_input` (disabled when no data context is selected or while processing).
- Processing flow:
  - If "Pipeline output": calls `process_chat_query()` → `get_intent_and_filters` + `retrieve_data` + `generate_final_response`.
  - If "csv upload": calls `handle_generic_csv()` for a more flexible, schema-agnostic query path.
- Supports three response types from the chatbot: plain string, `pd.DataFrame`, or a `dict` with `message` + `dataframe` keys.
- A "thinking" animation (CSS blinking dots) is shown while the bot is processing.

**Session state keys managed:**

| Key | Type | Purpose |
|-----|------|---------|
| `run_pipeline` | bool | Triggers pipeline execution |
| `pipeline_done` | bool | Marks pipeline completion |
| `final_df` | DataFrame | In-memory pipeline output |
| `elapsed` | float | Pipeline wall-clock time |
| `file_count` | int | Number of PDFs processed |
| `chat_history` | list | Full message log |
| `uploaded_df` | DataFrame | User-uploaded CSV |
| `uploaded_filename` | str | Tracks filename to detect re-uploads |
| `processing` | bool | Prevents double-send during bot response |

### `pipeline/` — Extraction Pipeline

A modular, sequential pipeline with one responsibility per module.

#### `parser.py`
- `extract_text_from_pdf(file_path)` — uses `pdfplumber` to read all pages and return raw text.
- `extract_chapter_4(file_path)` — isolates Chapter 4 content from the full document text, which contains the benefits/coverage information targeted for extraction.

#### `chunker.py`
- `chunk_text(text)` — splits long text into manageable chunks suitable for LLM context windows. Uses character or token count limits to avoid exceeding model input limits.

#### `llm_handler.py`
- `extract_from_chunk(chunk)` — sends a single chunk to an OpenAI model (via the `openai` library) with a structured extraction prompt. Returns a parsed dictionary or JSON object representing the benefit data in that chunk.

#### `extractor.py`
- `aggregate_results(results)` — merges the per-chunk extraction outputs into a single coherent structured object, handling deduplication and consolidation across chunk boundaries.

#### `formatter.py`
- `to_dataframe(structured_data, filename)` — converts the aggregated structured data into a `pandas.DataFrame`, tagging each row with the source filename for traceability.

#### `validator.py`
- `validate_df(df)` — applies validation rules to the final DataFrame (e.g., type coercion, null handling, column normalization) before it is saved to CSV.

---

### `chatbot/main.py` — Chatbot Module

Provides the natural language query interface over the extracted data.

#### `get_intent_and_filters(user_input, columns)`
- Takes the user's question and the list of available DataFrame columns.
- Calls an LLM to classify the intent (e.g., lookup, filter, aggregate, compare) and extract relevant filter parameters (e.g., specific benefit names, plan types, dollar amounts).
- Returns a structured intent object.

#### `retrieve_data(df, intent)`
- Applies the intent's filters to the DataFrame using pandas operations.
- Returns `(relevant_data, category)` — the filtered subset and a label for downstream response formatting.

#### `generate_final_response(user_input, relevant_data, category)`
- Takes the original question, the filtered data, and the category tag.
- Calls an LLM with the data context to generate a human-readable answer.
- Returns either a plain string or a dict with `message` + `dataframe`.

#### `handle_generic_csv(user_input, df)`
- A more flexible handler for arbitrary uploaded CSVs with unknown schemas.
- Inspects the DataFrame structure dynamically and routes the query appropriately.
- Designed to work without assumptions about column names or data types.

### `EOC / EOC2` — Input Data

- `./EOC` is the primary input folder scanned by the pipeline.
- `./EOC2` appears to be a secondary or alternate batch of EOC PDF documents (e.g., a different insurer or plan year).
- Place PDF files here before running the pipeline. The app counts and lists them in the UI.

---

### `output/` — Generated Output

- Created automatically on first run (`Path("./output").mkdir(parents=True, exist_ok=True)`).
- Stores `output.csv` — the concatenated result of all processed PDFs.
- This file persists between sessions and is reloaded automatically if the pipeline has not been re-run.

## Setup & Installation

### Prerequisites

- Python 3.9+
- An ollama and llm model (used by `llm_handler.py` and the chatbot)

### Steps

```bash
# 1. Clone the repository
git clone https://github.com/chaitanyatyagi/HealthworksAI-Assessment.git
cd HealthworksAI-Assessment

# 2. Create and activate a virtual environment (recommended)
python -m venv venv
source venv/bin/activate        # macOS/Linux
venv\Scripts\activate           # Windows

# 3. Install dependencies
pip install -r requirements.txt

```

## Usage Walkthrough

### 1. Batch Extract from EOC PDFs

1. Place one or more EOC PDF files in the `./EOC` directory.
2. In the app, the **Pipeline** section shows a count of detected PDF files.
3. Click **▶ Run Pipeline** to start the batch extraction.
4. Watch live per-file status badges and the elapsed-time counter.
5. On completion, metrics (files processed, rows, columns, time) are displayed.

### 2. Review & Download Extracted Data

1. In the **Data Context** section, select **Pipeline output**.
2. A scrollable preview table of the extracted CSV appears.
3. Click **↓ Download CSV** to save `output.csv` locally.

### 3. Query with the Chatbot

1. In **Data Context**, choose either **Pipeline output** or upload a CSV and select **csv upload**.
2. Type a question in the chat input (e.g., "What is the copay for a specialist visit?").
3. The assistant responds with a text answer and/or a filtered data table.

### 4. Upload an External CSV

1. Use the **Upload External CSV** file uploader.
2. Select **csv upload** as the data context.
3. Ask questions about your uploaded data — the chatbot handles arbitrary schemas.

### 5. Clear Output

- Click **🗑 Delete Output** to remove `output.csv` and reset the pipeline state.

## Key Design Decisions

**Threaded timer** — The elapsed-time counter runs in a daemon thread (`threading.Thread`) rather than blocking the main Streamlit execution. `add_script_run_ctx` is used to safely pass Streamlit's script context into the background thread, which is required for any thread that calls Streamlit APIs.

**Dual data context** — The app explicitly separates "pipeline output" from "uploaded CSV" because EOC data has a known, structured schema while uploaded CSVs are arbitrary. This separation allows the chatbot to use schema-aware retrieval for the former and a more generic approach (`handle_generic_csv`) for the latter.

**Stateless LLM calls** — Each pipeline chunk and each chatbot turn makes a fresh LLM API call. There is no persistent vector store or embedding index; retrieval is done directly against the in-memory DataFrame using pandas operations guided by LLM-extracted filters.

**Session state as source of truth** — All pipeline results, chat history, and UI state live in `st.session_state`, making the app fully resumable within a browser session without a database.
